# 重构安全指南 - 避免再次出现问题

## 🎯 核心原则

**在改动任何代码之前，先问自己三个问题：**
1. 这段代码有什么特殊的fallback逻辑吗？
2. 这个改动会影响文件加载顺序吗？
3. 我能立即测试这个改动吗？

---

## 📋 重构前检查清单

### 1. 代码风险评估

**🔴 高风险代码（谨慎改动）**
- [ ] UI层代码（`ui/*.js`）
- [ ] 战斗系统（`game/Battle.js`）
- [ ] 装备系统（`ui/equipNode.js`）
- [ ] 存档系统（`game/DataManager.js`）
- [ ] 初始化代码（`main.js`、`game.js`）

**🟡 中风险代码（小心改动）**
- [ ] 核心游戏逻辑（`game/*.js`）
- [ ] 数据配置（`data/*.js`）
- [ ] 交易/商店系统

**🟢 低风险代码（可以改动）**
- [ ] 工具函数（`util/*.js`）
- [ ] 纯配置数据
- [ ] 新增功能代码

### 2. Fallback逻辑识别

**在改动前，搜索文件中的这些模式：**

```javascript
// 模式1: || 运算符
var value = someFunction() || defaultValue;
var icon = getIcon(id) || "default.png";

// 模式2: 三元运算符
var result = condition ? value1 : value2;

// 模式3: 短路求值
var data = obj && obj.property;

// 模式4: 函数存在性检查
if (fn) { fn(); }
fn && fn();
```

**如果发现这些模式：**
- ❌ 不要用 `SafetyHelper.safeCall`
- ✅ 用 `SafetyHelper.safeCallWithFallback`
- ✅ 或者保持原样不动

### 3. 文件依赖检查

**如果要创建新文件或修改加载顺序：**

- [ ] 检查 `project.json` 或 `main.js` 的文件加载顺序
- [ ] 确认新文件在使用它的文件**之前**加载
- [ ] 优先使用**内联配置表**，避免外部依赖

**内联配置表模板：**
```javascript
// 在文件内部定义配置
var _configMap = {
    key1: value1,
    key2: value2
};

// 使用配置
function getConfig(key) {
    return _configMap[key];
}
```

## 🛠️ 工具使用指南

### SafetyHelper 使用决策树

```
需要调用函数？
├─ 有 || fallback 逻辑？
│  ├─ 是 → 用 safeCallWithFallback 或保持原样
│  └─ 否 → 继续
├─ 是UI层代码？
│  ├─ 是 → 保持原样，不要改
│  └─ 否 → 可以用 safeCall
└─ 是战斗系统？
   ├─ 是 → 保持原样，不要改
   └─ 否 → 可以用 safeCall
```

### 正确使用示例

```javascript
// ✅ 正确：没有fallback逻辑
var damage = SafetyHelper.safeCall(calculateDamage, 0, enemy);

// ✅ 正确：使用 safeCallWithFallback
var icon = SafetyHelper.safeCallWithFallback(
    itemConfig.getIcon,
    "icon_1.png",
    itemId
);

// ✅ 正确：UI层保持原样
var icon = itemConfig.getIcon(itemId) || "icon_1.png";

// ❌ 错误：破坏了fallback逻辑
var icon = SafetyHelper.safeCall(itemConfig.getIcon, "icon_1.png", itemId);
```

## 🚨 危险信号识别

### 改动后出现这些情况，立即回退：

1. **游戏闪退**
   - 症状：启动后立即退出
   - 可能原因：文件加载顺序、undefined引用
   - 处理：立即回退，检查依赖关系

2. **UI消失或异常**
   - 症状：界面不显示、按钮无响应
   - 可能原因：fallback逻辑被破坏
   - 处理：立即回退，检查 || 运算符

3. **功能失效**
   - 症状：某个功能无法使用
   - 可能原因：函数调用方式改变
   - 处理：立即回退，对比原代码

4. **新存档无法创建**
   - 症状：点击新建存档后闪退
   - 可能原因：初始化代码问题
   - 处理：立即回退，这是最严重的问题

## 📝 重构工作流程

- [ ] 评估风险等级

**第2步：准备**
- [ ] 备份原文件
- [ ] 记录当前功能状态

**第3步：改动**
- [ ] 只改一个地方

**第4步：测试**
- [ ] 立即启动游戏
- [ ] 测试相关功能
- [ ] 记录测试结果

**第5步：决策**
- ✅ 测试通过 → 提交改动，继续下一个
- ❌ 测试失败 → 立即回退，分析原因

## 🎓 经验教训总结

### 教训1：不要过度统一
**错误做法：**
"所有图标获取都用IconManager统一"
"所有函数调用都用SafetyHelper包装"

**正确做法：**
"新功能用新工具，旧代码保持原样"
"只在明确安全的地方应用工具"

### 教训2：内联优于外部
**错误做法：**
创建 `configManager.js`、`roleConfigTable.js`
→ 文件加载顺序问题 → 闪退

**正确做法：**
在 `role.js` 内部定义 `_roleConfigMap`
→ 零依赖 → 稳定

### 教训3：UI层要保守
**错误做法：**
"equipNode.js 的图标获取逻辑不统一，要改"
→ 精英手枪装备失败

**正确做法：**
"equipNode.js 能工作就不动，只改逻辑层"
→ 功能正常

### 教训4：战斗系统是红线
**错误做法：**
"Battle.js 也用SafetyHelper优化一下"
→ 战斗UI崩溃

**正确做法：**
"Battle.js 是核心系统，不要轻易改动"
→ 功能正常

## 🔄 快速回退指南

### Git 回退（如果使用版本控制）
```bash
# 查看改动
git diff

# 回退单个文件
git checkout -- path/to/file.js

# 回退所有改动
git reset --hard HEAD
```

### 手动回退（如果有备份）
```bash
# 恢复备份
cp file.js.backup file.js
```

### 紧急回退（如果没有备份）
1. 从 BUG_SUMMARY.md 或 REFACTOR_LOG.md 找到原始代码
2. 手动恢复关键部分
3. 重新测试

## 📊 重构优先级矩阵

| 风险 | 收益高 | 收益低 |
|-----|-------|-------|
| **高** | ⚠️ 谨慎评估 | ❌ 不要做 |
| **中** | ✅ 可以做 | ⏸️ 暂缓 |
| **低** | ✅✅ 优先做 | ✅ 可以做 |

**示例：**
- role.js 内联配置表：低风险 + 高收益 = ✅✅ 优先做
- Battle.js 用SafetyHelper：高风险 + 低收益 = ❌ 不要做
- equipNode.js 统一图标：高风险 + 中收益 = ⚠️ 谨慎评估（结果：失败）

---

**记住：重构的目标是改进代码，而不是破坏功能。当不确定时，保持原样总是最安全的选择。**
